# Knowledge Domain

The `Knowledge` domain represents static, reusable reference material that grounds the AI and improves `Decision` quality without serving as a source of authority.

## 1. Purpose
While `Memory` represents experiential context generated by the system's own `Reflections`, `Knowledge` represents explicit, external facts provided to the system. It encompasses API documentation, runbooks, codebase schemas, and architecture definitions. It provides the AI with the necessary rules of engagement for a specific environment.

## 2. Ownership
- **Parent**: `Knowledge` can be bound globally (System Knowledge) or scoped specifically to a `Workspace` (Workspace Knowledge).
- **Children**: None.

## 3. Responsibilities
- Storing static markdown documents, code snippets, and reference URLs.
- Making reference material queryable via Semantic Search (RAG) during the `Mission` planning phase.
- Supporting the AI Provider Router by injecting relevant facts into the context window.

## 4. Lifecycle
- **Indexed**: The documentation has been ingested, chunked, and embedded into the system.
- **Active**: The knowledge is actively available for RAG queries.
- **Stale**: The original source document has been modified externally, requiring a re-index.
- **Deleted**: The knowledge is removed from the active index.

## 5. Invariants
- `Knowledge` is purely advisory. It supports `Decisions` but is NEVER the source of authority (the `Operator` and the `Mission` state hold authority).
- `Knowledge` MUST remain strictly separate from `Memory`.

## 6. Relationships
- **Workspace**: Often owns a localized set of `Knowledge` documents.
- **DecisionPlan**: Uses `Knowledge` to improve the accuracy of the proposed execution steps.
- **Memory**: Overrides `Knowledge` if an experiential heuristic conflicts with static documentation.

## 7. Success Criteria
- The system correctly surfaces the relevant API endpoints or runbook steps when the AI attempts to solve a `Decision` related to those topics, reducing hallucination rates.

## 8. Examples
- **Knowledge**: The official Stripe API documentation markdown file.
- **Knowledge**: A company's internal "Incident Response Playbook.pdf".

## 9. Future Extensibility
- **Automated Sync**: Connecting the `Knowledge` domain to external knowledge bases (like Notion or Confluence) to automatically re-index when upstream docs change.
- **Graph Relationships**: Mapping `Knowledge` chunks into a knowledge graph to understand dependencies (e.g., "Service A depends on Service B").

## 10. Out of Scope
- Storing live execution state or dynamic variables.
- Storing lessons learned from past failures (this belongs to `Memory`).
